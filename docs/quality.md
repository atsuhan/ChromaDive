# Quality Rules

## 完了条件
- テストが通るだけでは完了ではない。実際にアプリ上で動作確認できること
- モック・スタブ・ダミーデータで誤魔化して完了扱いにしない
- 「コードが存在する」は不合格。「動く」が合格

## テスト
- 新しい機能にはテストを書く
- 既存テストを壊さない。変更後は必ずテスト実行
- テストは実際のデータフローで検証する（モックは外部依存のみ）

## コミット前チェック
- lint / format が通ること
- 型チェックが通ること
- 既存テストがすべてパスすること

## コードレビュー観点
- 関数が1つの責務に集中しているか
- コメントが「なぜ」を説明しているか
- 命名から処理内容が推測できるか
- エラーハンドリングが適切か
